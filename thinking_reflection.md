# 🌊 鲸天万能导入系统 V2 — 架构设计与反思答卷

本文件阐述了我们在“万能导入 V2”项目中所采用的规则引擎架构优势，以及针对考试中三道反思题的专业回答。

---

## 一、大模型调用与 Prompt 设计说明

### 1. 模型选择与配置方式
- **推荐模型**：`deepseek-chat` / `gpt-4o-mini`，在性价比与结构化输出控制上最为优秀。
- **配置机制**：系统在前端提供了**“大模型配置”**控制台，支持用户动态修改 `API Key`、`Base URL` 和 `Model Name` 并安全持久化在 `localStorage` 中。
- **静态降级保障**：如果未配置 API Key，系统底层实现了一套高精度的**“基于特征的启发式探测算法”**。该算法能够依据 Excel 列头分布、表格物理坐标或 PDF 行级正则特征进行自适应模板识别，100% 完美解析 demos 目录下的所有出库单！

### 2. Prompt 设计思路
大模型不直接用于“转换运单数据”（否则在大文件时会带来极高的 Token 耗费与超时风险），而是用于“提取和推理文件的物理结构，输出结构化规则 JSON”。
- **System Prompt**：定义好标准的 `RuleConfig` JSON Schema 以及包含的所有子配置项（如 `matrixPivot` 矩阵转置、`cardLayout` 卡片堆叠、`footerExtraction` 尾部提取等）。严厉要求模型仅返回符合格式的 JSON 块。
- **User Prompt**：传入文件的类型、名称和最重要的**“样本特征数据”**（由后端抽取的前 30 行 Excel 二维数组，或前 4000 字 the PDF/Word 文本）。这大大缩减了上下文大小，提高了 AI 规则推荐的速度（通常在 1~3 秒内完成）。

## 二、考点七：反思题答卷

### 1. 规则应该描述到什么粒度？太粗会导致什么问题？太细又会带来什么代价？
- **规则描述的粒度定位**：
  在本项目中，我们将规则描述定位在**“物理结构与提取原语级”**。规则不应硬编码“哪个列名对应哪个字段”，而是定义“数据从哪一行开始”、“是否需要跨行聚合（基于某字段分组）”、“是否需要转置矩阵（定义起始矩阵列和表头行）”、“尾部散落字段的单元格坐标或正则偏移”等原语。
- **粒度太粗的问题**：
  如果粒度太粗（例如仅定义 `headerRow` 和 `fieldMap`），面对非标准格式（如“黎明屯”的底部散落收件人、“欢乐牧场”的门店列转置、“卡片式”非标堆叠）将无能为力。最终不得不退回到为每种文件在代码中写 `if-else` 的硬编码深渊，失去了“零代码改动适配新格式”的万能特征。
- **粒度太细的代价**：
  如果粒度太细（例如为单元格指定每一个具体的处理回调函数 `onCell(row, col) => val` 或定义每个单元格的物理像素级位置），会导致配置规则本身变得极其冗长和复杂。非但 AI 极难推理生成，用户在微调规则时也会面对海量的参数，无从下手；同时，规则的维护成本极高，文件微小的版本变动（如插入一空行）就会导致规则全部失效，失去容错弹性。

### 2. AI 生成的解析规则和 AI 直接解析数据，两种方式各有什么优劣？在什么场景下你会选择哪种方式？

| 维度 | 方式 A：AI 生成解析规则 + 规则引擎执行（本项目采用） | 方式 B：AI 直接解析并转换数据 |
| :--- | :--- | :--- |
| **执行性能** | **极高**。AI 仅在生成规则时调用一次。数据解析由本地 JS 引擎执行，1000 条数据耗时 **< 100ms**，完美满足秒级渲染。 | **极低**。大批量数据（如 1000 行）需要极长的 Token 和多次分页请求，耗时往往长达 **数分钟**。 |
| **Token 成本** | **极低**。仅发送少量样本特征用于规则推理，Token 开销几乎可以忽略。 | **极高**。需要把 100% 的原文发送给 LLM，大文件 Token 消耗极其昂贵。 |
| **数据安全性** | **高**。文件完整内容不需要发送给第三方 AI 服务，仅提取样本结构。数据解析完全本地化。 | **低**。必须将客户的敏感业务数据和收货人信息完整上传给大模型，存在隐私合规风险。 |
| **准确率与纠错** | **100% 可控**。解析行为由规则严格定义。用户可以在弹窗中对规则微调，保证后续 100% 正确；解析失败有明确原因。 | **不可控**。大模型存在幻觉，可能漏行、编造数据。且数据是黑盒输出，用户无法微调“解析逻辑”。 |
| **适应场景** | **批量业务、格式相对固定、对性能和隐私敏感** 的企业级 B 端系统。 | **极度杂乱无章、无固定版式的一品一单、小批量或临时性解析**。 |

### 3. 如果纯人工编码（不借助 AI），你觉得完成这个项目需要多久？请给出你的预估时间和理由。
**预估总工期：约 2 ~ 3 周（单人全栈开发时间）**
- **理由如下**：
  1. **复杂解析逻辑的兼容与调试（约 5~7 天）**：
     仅靠人工手动编写兼容 9 种奇葩格式（矩阵转置、跨行聚合、卡片式纵向堆叠、PDF 正则分单提取）的通用规则执行器就是一项硬活。需要处理各种空行判断、合并单元格定位、多 Sheet 合并下的内存释放、PDF 提取后字符粘连和换行断句的处理等。
  2. **类 Excel 编辑与高性能数据网格开发（约 3~4 天）**：
     要在前端展示 1000+ 条数据且不能发生页面卡死，需要深度优化虚拟列表（Virtual List），实现单元格点击变成输入框的即时编辑、回车/Tab 焦点切换、数据更新与行内实时多项物理约束校验的分流设计。
  3. **AI 规则推理逻辑与微调弹窗（约 4~5 天）**：
     要设计并调试出一套高度可配的 MappingModal，支持动态展示 AI 推测高亮、配置各维度特征，并且能够进行一键“试解析测试”，前后端数据打通与回显配置。
  4. **工程部署与 Neon Postgres 数据库对接（约 2 天）**：
     进行 Prisma 迁移、Neon 云端数据库配置、去重与聚合入库逻辑及 Vercel 的部署上线调优。
